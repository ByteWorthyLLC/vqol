# Phase 4 Context: Repo Virality Through Curiosity

## Goal

Make vqol public-ready and weird enough to spread through intellectual curiosity, not promotion.

## Current Baseline

Already present:

- Working Svelte PWA.
- Local-only storage.
- Practice config.
- Four locale files.
- Chart, print export, reminders, PWA shell.
- Public repo hygiene files.
- CI/deploy scaffolding.
- Telemetry audit.

Now implemented locally:

- Seeded demo mode.
- Curiosity lab.
- Local-first proof panel.
- Runtime offline-readiness inspector.
- One-file fork audit.
- Printable QR poster generator.
- Reference-only / permissioned / bring-your-own instrument mode.
- Calendar follow-up export.

Still missing:

- Public GitHub remote.
- Live GitHub Pages demo.
- Demo GIF/screenshots/social preview.
- Curiosity-first HN/Reddit/LinkedIn drafts.
- Real-device verification.
- LSHTM permission outcome or reference-only mode.

## Amplification Thesis

The repo should spread because it is a strange but credible artifact:

> A patient outcomes workflow that runs from static files, stores data locally, has no account system, no telemetry, and can be clinic-branded with one JSON file.

The weirdness must sit in demo, proof, and lab layers. The patient-facing clinical flow stays calm, literal, and non-gamified.

## Intellectual Curiosity Rule

Promotion is not the point. The project should be interesting because it demonstrates:

- A clinical-adjacent outcomes workflow without a backend.
- Patient-owned browser-local health data.
- One-file practice white-labeling.
- Offline-capable score history.
- A real instrument-licensing constraint handled as architecture, not hand-waving.

Channel copy comes after those artifacts exist.

## Safety Boundary

Any amplification feature must preserve:

- No clinical interpretation.
- No severity labels.
- No patient comparison or competition.
- No default public sharing of score/condition data.
- No telemetry.
- No unlicensed instrument text/constants.
