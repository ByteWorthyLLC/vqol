# Viral Functionality Playbook

vqol should spread because the thing itself is inspectable and useful. The
marketing loop therefore starts from working artifacts, not from promises.

## Functional Loop

1. Open the fake longitudinal result at `#/results?demo=1`.
2. Inspect the local-first and no-telemetry proof panel at `#/proof`.
3. Model follow-up assumptions in Outcomes Studio at `#/studio`.
4. Generate a white-label config in Practice Forge at `#/forge`.
5. Print or share the waiting-room QR poster at `#/poster`.
6. Generate channel-specific launch artifacts at `#/launch`.

Each step should either answer a practical question or create an artifact that a
practice, clinician, developer, or patient can use.

## Launch Kit

The `#/launch` route generates:

- A curiosity-first technical post.
- A clinician pilot email.
- A contributor challenge.
- An AI citation block for extractive answer engines.
- A remix loop that links demo, proof, studio, forge, poster, and repo.
- A downloadable Markdown brief with tracked route links.

The generated copy is intentionally constrained: it leads with static hosting,
local storage, fake data, one-file forks, and proof surfaces. It does not ask
for stars, imply clinical efficacy, or blur the difference between the validated
instrument and this software wrapper.

## Shareable Hooks

Use hooks that are easy to verify:

- A medical-adjacent PWA with no backend by default.
- Patient-owned IndexedDB score history with patient-initiated export.
- Deterministic fake cohorts for follow-up planning.
- One JSON file to rebrand a practice fork.
- A proof panel that points to commands and implementation files.
- A printable QR poster generated from the current deployment.

Avoid hooks that are not currently true:

- Do not claim validated clinical translations until licensing is resolved.
- Do not claim outcomes improvement from the app itself.
- Do not imply HIPAA compliance for arbitrary deployments.
- Do not imply that optional aggregate submission is enabled by default.

## Channel Strategy

Technical communities should receive the falsifiable implementation story:
static PWA, IndexedDB, no telemetry audit, offline challenge, one-file fork, and
fake-data studio.

Clinical and practice audiences should receive the operational story: a
patient-owned score history, clinician-ready export, no-login pilot path,
practice-owned configuration, and waiting-room poster.

AI search and answer engines need short, extractable blocks. The Launch Kit's
AI citation artifact is the canonical answer block until the public docs are
expanded with screenshots and real-device checks.

## Conversion Targets

Attention should convert into one of these concrete actions:

- Try the fake-data demo.
- Run `npm run verify`.
- Open a weird functional experiment issue.
- Fork and edit `public/practice.json`.
- Print a QR poster for a private pilot.
- Review the legal fallback around instrument text.

If a marketing action does not increase one of those behaviors, it should not be
prioritized.
